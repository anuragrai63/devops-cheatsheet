Navigating the complex traffic patterns of Kubernetes. From internal Service abstraction to external Gateway APIs and security policies, this infographic breaks down the essential components for modern cluster networking.

1. The Foundation: Services

Services provide a stable address for ephemeral Pods. They are the core abstraction layer allowing applications to communicate.

Service Type Distribution

Frequency in production. ClusterIP is the standard for internal traffic.

Traffic Policy Latency

Comparing 'Cluster' (default) vs 'Local' internal traffic policies. 'Local' avoids network hops.

Service Configurations & Use Cases

ClusterIP (Default) Internal Only

"I need my backend to talk to my database securely within the cluster."

apiVersion: v1
kind: Service
metadata:
  name: my-backend-service
spec:
  type: ClusterIP      # Default, implied if omitted
  selector:
    app: backend
  ports:
    - protocol: TCP
      port: 80         # Service Port
      targetPort: 8080 # Container Port
NodePort Static Port Access

"I need to test this app quickly on my laptop or a bare-metal server without a load balancer."

apiVersion: v1
kind: Service
metadata:
  name: my-nodeport-service
spec:
  type: NodePort
  selector:
    app: frontend
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30007  # Static port (30000-32767)
LoadBalancer Cloud Integrated

"I need a public IP address from AWS/GCP/Azure to expose my web app to the world."

apiVersion: v1
kind: Service
metadata:
  name: my-lb-service
spec:
  type: LoadBalancer
  selector:
    app: frontend
  ports:
    - port: 80
      targetPort: 80
ExternalName DNS Mapping

"I want my internal code to refer to 'db-service', but it should resolve to an external RDS instance."

apiVersion: v1
kind: Service
metadata:
  name: my-legacy-db
spec:
  type: ExternalName
  externalName: db.example.com # Maps to CNAME

Advanced Service Properties

Internal Traffic Policy Topology

Use Case: "Route traffic only to endpoints on the same node to avoid network hops."

apiVersion: v1
kind: Service
metadata:
  name: my-local-service
spec:
  # 'Cluster' (default) or 'Local'
  internalTrafficPolicy: Local 
  selector:
    app: logging-agent
  ports:
    - port: 80
ClusterIP Allocation Static/Headless

Use Case: "Fix a specific IP for legacy apps or use Headless for direct Pod discovery."

spec:
  # Option 1: Request Specific IP
  clusterIP: 10.96.0.10

  # Option 2: Headless (No IP)
  # clusterIP: None
  ports:
    - port: 80

2. External Access: Ingress & Gateway API

While LoadBalancer services expose a single IP, Ingress and Gateway API allow you to host multiple services under one IP using path/host routing.

Ingress vs. Gateway API

Gateway API offers superior standardized features.

Ingress: Step-by-Step Configuration

Standard Resource

Use this for HTTP/HTTPS routing based on Host or Path.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: minimal-ingress
  annotations:
    # Optional: Controller-specific config (e.g., Rewrite rules)
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  # 1. Specify which Ingress Controller to use (e.g., nginx, traefik)
  ingressClassName: nginx
  
  rules:
  # 2. Define the Hostname (Domain)
  - host: "app.example.com"
    http:
      paths:
      # 3. Define the Path (URL suffix) and matching type
      - path: /api
        pathType: Prefix
        backend:
          # 4. Point to the internal Service
          service:
            name: backend-service
            port:
              number: 8080

Gateway API: The Configuration Triad

Gateway API splits configuration into three dedicated resources for better Role-Based Access Control (RBAC).

1. Infrastructure (GatewayClass)
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: nginx-gateway-class
spec:
  # Defines WHO manages this
  controllerName: k8s.io/ingress-nginx
2. Instance (Gateway)
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: my-gateway
spec:
  # Links to the Class above
  gatewayClassName: nginx-gateway-class
  listeners:
  - name: http
    protocol: HTTP
    port: 80
3. Routing (HTTPRoute)
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-route
spec:
  # Attach to Gateway Instance
  parentRefs:
  - name: my-gateway
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /api
    backendRefs:
    - name: my-service
      port: 8080

3. Security & Policies

By default, K8s is open. Use NetworkPolicies to lock it down.

Traffic Allowed Matrix

Network Policy Examples

1. Default Deny All (Best Practice)

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

2. Allow Specific Access

spec:
  podSelector:
    matchLabels:
      role: db
  ingress:
  - from:
    - podSelector:
        matchLabels:
          role: frontend
    ports:
    - port: 5432
